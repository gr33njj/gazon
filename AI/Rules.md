# Rules of Engagement — для любого AI-агента в этом проекте

Эти правила читает Claude Code (или любой другой агент) перед тем, как что-то менять в `GameProject/`.
Конкретные инструменты (Unity MCP, Project MCP и т.д.) появятся по мере фаз из `Docs/Architecture.md` —
правила ниже актуальны уже сейчас, даже без единого MCP-сервера.

1. **Никакого прямого удаления ассетов/сцен без подтверждения человека.** Если задача требует убрать
   файл сцены, префаб или запись в БД — предложи это явно и дождись подтверждения, не делай молча.
2. **SQLite прежде всего.** Как только `Database/project.sqlite` станет использоваться реальным кодом:
   сначала регистрируй игровую сущность (тип клиента, реплику, апгрейд, штраф) в БД, только потом
   генерируй C#/ScriptableObject, который на неё ссылается. Нельзя заводить контент, дублирующий то,
   что уже есть в БД, — сначала `query_knowledge_base`-эквивалент (пока просто SELECT), потом создание.
3. **Unity Editor однопоточный.** Когда появится Unity MCP-мост — все вызовы к Editor API должны
   заходить через очередь на главный поток (`EditorApplication.update`), никаких прямых вызовов
   из фонового потока Python-сервера.
4. **GDD — источник истины по механикам, `Decisions.md` — источник истины по тому, почему что-то
   поменялось.** Если меняешь баланс/контент относительно `Docs/GDD.md` — обнови и GDD, и добавь
   запись в `Decisions.md`. Рассинхрон между кодом и GDD — это то, чего нужно избегать в первую очередь.
5. **Ветки и коммиты — по человеку.** Не пуш и не коммить без явного запроса; агент может подготовить
   изменения в рабочем дереве, финальное решение коммитить/пушить — за человеком, который попросил задачу.
6. **Тон игры — часть спецификации, не декорация.** Чёрный юмор про прекарный труд в ПВЗ, абсурдные
   штрафы, издевательский донат — при любой генерации контента (реплики, штрафы, мемы) сверяйся
   с существующим тоном в `Docs/GDD.md`, а не изобретай нейтральный.
