## 2026-07-16 — Стартуем с HTML/Three.js MVP как источника GDD, портируем в Unity 6

Прототип `gamedev/GAZON PVZ Simulator.dc.html` уже содержит полный игровой цикл, экономику и тон игры.
Вместо того чтобы придумывать GDD с нуля, он извлечён в `Docs/GDD.md` построчным разбором кода MVP.
Дальнейшие геймдизайн-решения (новый контент, изменения баланса) фиксируются здесь, а не молча
меняются в коде — иначе `GDD.md` разойдётся с реальностью и агенты будут работать по устаревшему документу.

## 2026-07-16 — Порядок инфраструктуры: сначала Unity-прототип, потом MCP-серверы, потом веб-панель

Полная схема (Unity MCP + SQLite MCP + Docs MCP + Git MCP + веб-панель на ai.gr33njj.dev) не строится
одним махом — см. `Architecture.md`. Причина: до того как в Unity есть играбельный вертикальный срез,
неизвестно, какие из MCP-инструментов реально нужны в первую очередь, а какие можно отложить или не делать
вообще. Строим по мере боли, а не по чеклисту.

## 2026-07-16 — C#-скелет вертикального среза написан заранее, без Unity Editor на этой машине

`Assets/Scripts/` содержит рабочую (по коду, не проверенную компиляцией) реализацию цикла
докстанция → полки → окно выдачи → таймер смены — без QTE/мини-игр/телефона/штрафов/кражи/перекура
(сознательно, см. `Assets/Scripts/README.md`). Причина писать код до появления самого Unity-проекта:
C# — обычный текст, Unity сама сгенерирует `.meta` при первом импорте, а откладывать код до момента,
когда кто-то поставит Unity Hub, значило бы просто терять время параллельно с установкой.
**Важно:** ничего из этого не скомпилировано и не запущено в реальном Unity Editor — первое, что
нужно сделать после переноса папки в Unity-проект, это дать ему собраться и проверить на ошибки.
