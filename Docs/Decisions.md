`Assets/Scripts/` содержит рабочую (по коду, не проверенную компиляцией) реализацию цикла

**Важно:** ничего из этого не скомпилировано и не запущено в реальном Unity Editor — первое, что
нужно сделать после переноса папки в Unity-проект, это дать ему собраться и проверить на ошибки.

## 2026-07-16 — Unity MCP: не пишем свой мост, ставим готовый (MCP for Unity)

Изначально в `AI/MCP.md`/`Docs/Architecture.md` был план писать свой `HttpListener`-мост в Unity Editor
с нуля (Фаза 2, пункт 3). Проверил через веб-поиск: есть готовый, поддерживаемый, MIT-лицензированный
инструмент **MCP for Unity** (github.com/CoplayDev/unity-mcp), работает на Unity 2021.3 LTS–6.x,
из коробки поддерживает Claude Code. Причина отказаться от самописного моста: нет смысла хрупко
переизобретать то, что уже есть и поддерживается сообществом — риск багов в самописной версии выше,
а готовое решение сразу закрывает сегодняшнюю боль (ручная пересылка ошибок консоли туда-сюда между
Unity Editor и чатом с агентом). Ограничение то же, что было бы и у самописного моста: агент должен
работать на той же машине, что и открытый Unity Editor — сессия на VPS физически не видит чужой
localhost, так что для реального использования MCP нужен Claude Code, запущенный локально рядом
с Unity, а не эта сессия.

## 2026-07-16 — Порядок реализации ai.gr33njj.dev: раннер → GitHub Issues → панель

Договорились не строить веб-панель первой. Порядок: (1) выделенная машина, на которой постоянно
включён Unity Editor с открытым проектом — обязательное условие для MCP for Unity, который требует
живой редактор, а не headless-процесс; (2) GitHub Issues/Projects как бесплатный общий трекер задач
для двоих, без всякой своей инфраструктуры; (3) сама панель — только после того, как первые два пункта
реально в ходу и понятно, чего конкретно не хватает поверх голого GitHub. Причина: панель без раннера
не может ничего реально выполнять, а голый GitHub закрывает 80% ценности «делать запросы вместе и
видеть процесс» бесплатно и сразу.

## 2026-07-17 — Связь ПК-раннера и VPS: Tailscale, не port forwarding

Домашний ПК без белого IP не может принимать входящие соединения от VPS напрямую. Вместо проброса
порта на роутере (дыра в сеть, ломается при смене IP) — Tailscale: mesh-VPN, где оба узла делают
исходящее соединение наружу, что всегда проходит через NAT. Установлен на ПК пользователя и на этой
VPS (`gr33njj-vps`). Авторизация ВПС в тейлнет — через ссылку `tailscale up` печатает в консоль,
подтверждается в браузере на аккаунте пользователя (я не могу пройти этот шаг сам, только человек).
Подробности и следующие шаги (локальный процесс-приёмник задач на ПК, WebSocket для совместной работы,
WebGL-превью в браузере) — в `Docs/Architecture.md`, раздел «Связь VPS ↔ домашний ПК».

## 2026-07-17 — Отказ от входящих соединений к ПК-раннеру: только исходящий опрос

Пробовали входящий SSH (VPS → ПК) как ранний прототип запуска задач для будущей панели. Уткнулись
в реальную проблему: на ПК-раннере постоянно работает AmneziaVPN (нужен пользователю для доступа
к Claude/Anthropic из его региона), и он, похоже, глушит входящие TCP-подключения на Tailscale-интерфейсе
на уровне драйвера — при том что Windows Firewall и служба sshd настроены верно, а сам Tailscale
(другой протокол) пингуется нормально. Пользователь не готов отключать Amnezia даже на секунду теста —
это разорвёт его связь с Claude Code. Решение: не чинить это, а поменять архитектуру — раннер должен
сам инициировать исходящие соединения к VPS (как self-hosted раннер GitHub Actions), а не принимать
входящие. Исходящие уже надёжно работают (иначе не работал бы и сам Claude Code через Amnezia).
Подробности — `Docs/Architecture.md`, раздел «Связь VPS ↔ домашний ПК».

## 2026-07-17 — Панель v1 проверена на реальной задаче + SceneBuilder стал источником истины для CI

Первый end-to-end прогон панели (issue → agent-task.yml → Claude Code → коммит в main → webgl-build)
вскрыл цепочку мелких проблем на раннере, все исправлены:
- `claude` CLI не был установлен как отдельный бинарник (пользователь работал только через расширение
  VSCode и десктоп-приложение) — поставлен `npm install -g @anthropic-ai/claude-code`, залогинен разово
  интерактивно.
- `node`/`npm` и `gh` CLI на раннере не было вовсе — поставлены через `winget`.
- После любой установки, меняющей PATH, раннер (`run.cmd`) нужно перезапускать вручную — процесс читает
  PATH один раз при старте, обновления не подхватывает.
- Windows PowerShell 5.1 ломает non-ASCII (кириллицу) в сгенерированных из `run:`-блоков `.ps1`-файлах —
  без BOM он читает их в системной кодировке, кириллица в квотированных строковых литералах превращается
  в мусор и рвёт парсинг («missing terminator»). Комментарии (`#...`) и YAML-метаданные (`description:`)
  не страдают — проблема только в quoted-строках внутри `run:`. Убрали кириллицу из единственной такой
  строки в agent-task.yml.
- `GITHUB_TOKEN` по умолчанию read-only на приватном репозитории — `git push` от `actions/checkout`
  падал 403. Добавлено `permissions: contents: write` на job в agent-task.yml.

После починки всей цепочки агент реально доделал задачу (issue #14, стеллажи 24 ячейки вместо 3) и
запушил рабочий код (commit `feebbe2`) — но WebGL-превью всё равно не обновилось. Причина отдельная и
архитектурная: `BuildScript.cs` собирал уже сохранённую `.unity`-сцену как есть, а `SceneBuilder.cs` —
Editor-only меню-команда, которая требует ручного нажатия в открытом Editor и `Ctrl+S`; правки кода
генератора сцены сами по себе на уже собранную сцену не влияют. Headless-агент физически не может
воспроизвести этот шаг. Решение: `BuildWebGLCI()` теперь перед сборкой сам создаёт чистую временную
сцену, вызывает `SceneBuilder.BuildScene()` и сохраняет её поверх сцены из `EditorBuildSettings` —
`SceneBuilder.cs` стал единственным источником истины для CI-сборки WebGL-превью. Побочный эффект:
любые ручные правки `.unity`-сцены в Editor, не перенесённые в код `SceneBuilder.cs`, будут
перезаписаны следующим CI-прогоном — при ручной локальной доработке сцены переносить изменения в код
генератора, а не оставлять только в сохранённом файле сцены.

## 2026-07-20 — Полный порт MVP в Unity одним заходом

Пользователь прикрепил актуальный `gamedev/pvz_simulator_mvp.html` (не тот, что был на диске раньше —
старый `GAZON PVZ Simulator.dc.html` оказался файлом другого формата, с чужой обвязкой `support.js`/
`<x-dc>`/`<helmet>`, не сырым Three.js-исходником; `Docs/GDD.md` и `Database/project.sqlite` уже были
извлечены именно из содержимого нового файла, так что рассинхрона в контенте не было — только ссылка
на файл в `.claude/CLAUDE.md`/`AI/Context.md` была обновлена на `pvz_simulator_mvp.html`) и попросил
перенести всё разом, а не по одному issue за раз, как планировалось раньше.

Перенесено целиком: полная комната 1:1 (стены, проёмы, стойка, 2 окна выдачи, 24 ячейки на 3
стеллажах, докстанция, примерочная, курилка, стол возвратов, вход/выход — координаты в новом
`Core/RoomLayout.cs`, общий источник для `SceneBuilder` и рантайм-скриптов), архетипы клиентов
(обычный/бабушка/шопоголичка со взвешенным роллом), очередь (клиенты стоят в очереди, если оба окна
заняты, до 8 одновременно), примерочная (28% обычных / гарантированно шопоголички, 4-6 коробок
возврата), мини-игры «крокодил» и «Глаз Бога» (гейтят не выдачу, а получение коробки с полки —
уточнено чтением исходника: `orderedBy.guessed` проверяется в промпте коробки, не в окне выдачи),
QTE при выдаче (30% шанс), телефон (рилсы + донат-магазин: мат/микрофон/тройка/продажа чехлов/ирл-
концовка), кража (X, шанс поимки зависит от количества клиентов в зале), периодические абсурдные
штрафы (16-30 сек, 35% шанс) и штраф за телефон при клиентах (раз в сек, 10% шанс) — оба изначально
пропустил при первом проходе, добавил после сверки с GDD, курьер-конфронтация (спавнится при хрупкой
коробке в партии, таймаут ожидания увеличен с ~3.6 сек MVP до 25 сек — там это было завязано на
косметическую анимацию газели, которую не переносили), перекур (4 сек, кулдаун 45 сек, во время
перекура терпение клиентов у окна тает вдвое быстрее), ночная инвентаризация, увольнение/ребрендинг
VILBERIS, ирл-концовка за 400 000 ₽.

Контент (реплики/штрафы/мемы/товары бабушки/звания/архетипы) не хардкожен — экспортируется из уже
заполненной `Database/project.sqlite` в `Assets/Resources/GameContent.json` скриптом
`Tools/ai_suite/export_content.py`, читается `ContentDatabase.cs` через `Resources.Load` (в проекте
нет sqlite-плагина для Unity/WebGL, а правило AI/Rules.md «SQLite прежде всего» уже соблюдено — база
была заполнена в Phase 0).

Весь новый UI (HUD/телефон/панели мини-игр/экраны меню-паузы-итогов-увольнения-ирл) — на OnGUI
(IMGUI), тем же способом, каким был сделан старый `DebugHud.cs` (удалён, полностью заменён): в
проекте нет TextMeshPro/uGUI, добавлять пакет вслепую без Unity Editor под рукой не стали — рискованно
для CI. Полноценный визуальный UI на Canvas — по-прежнему открытая задача на будущее, если понадобится
именно красивый интерфейс, а не только рабочий.

Сознательные упрощения (см. также шапку `SceneBuilder.cs`): не портированы декоративные мелочи —
текстуры вывесок, шашечный пол, конус/куст-декорации, анимация подъезжающей газели и "полёта" коробок
с неё (коробки партии появляются на докстанции сразу). Столкновения со стенами/стеллажами/стойкой
теперь отдаёт `CharacterController` + `BoxCollider` — собственный ручной AABB-движок коллизий из
MVP не нужен, так как в Unity есть настоящая 3D-геометрия с коллайдерами.

Сохранена намеренная шутка MVP: 3D-мир (вывеска, стойка) не перекрашивается после ребрендинга в
VILBERIS — `SceneBuilder` не знает про `Purple`-флаг вообще, перекрашивается только HUD/меню в
`GameUI.cs`. Это не баг, а прочитанный в исходнике осознанный момент («Стойка осталась синей:
ребрендинг не завезли»).

Не проверено интерактивно (нет Unity на этой машине) — компиляция проверяется через CI
(`webgl-build.yml`), но реальную играбельность/баланс должен проверить человек в Editor или через
WebGL-превью на ai.gr33njj.dev/preview/.
