## 2026-07-16 — Unity MCP: не пишем свой мост, ставим готовый (MCP for Unity)

Изначально в `AI/MCP.md`/`Docs/Architecture.md` был план писать свой `HttpListener`-мост в Unity Editor
с нуля (Фаза 2, пункт 3). Проверил через веб-поиск: есть готовый, поддерживаемый, MIT-лицензированный
инструмент **MCP for Unity** (github.com/CoplayDev/unity-mcp), работает на Unity 2021.3 LTS–6.x,
из коробки поддерживает Claude Code. Причина отказаться от самописного моста: нет смысла хрупко
переизобретать то, что уже есть и поддерживается сообществом — риск багов в самописной версии выше,
а готовое решение сразу закрывает сегодняшнюю боль (ручная пересылка ошибок консоли туда-сюда между
Unity Editor и чатом с агентом). Ограничение то же, что было бы и у самописного моста: агент должен
работать на той же машине, что и открытый Unity Editor — сессия на VPS физически не видит чужой
localhost, так что для реального использования MCP нужен Claude Code, запущенный локально рядом
с Unity, а не эта сессия.

## 2026-07-16 — Порядок реализации ai.gr33njj.dev: раннер → GitHub Issues → панель

Договорились не строить веб-панель первой. Порядок: (1) выделенная машина, на которой постоянно
включён Unity Editor с открытым проектом — обязательное условие для MCP for Unity, который требует
живой редактор, а не headless-процесс; (2) GitHub Issues/Projects как бесплатный общий трекер задач
для двоих, без всякой своей инфраструктуры; (3) сама панель — только после того, как первые два пункта
реально в ходу и понятно, чего конкретно не хватает поверх голого GitHub. Причина: панель без раннера
не может ничего реально выполнять, а голый GitHub закрывает 80% ценности «делать запросы вместе и
видеть процесс» бесплатно и сразу.

## 2026-07-17 — Связь ПК-раннера и VPS: Tailscale, не port forwarding

Домашний ПК без белого IP не может принимать входящие соединения от VPS напрямую. Вместо проброса
порта на роутере (дыра в сеть, ломается при смене IP) — Tailscale: mesh-VPN, где оба узла делают
исходящее соединение наружу, что всегда проходит через NAT. Установлен на ПК пользователя и на этой
VPS (`gr33njj-vps`). Авторизация ВПС в тейлнет — через ссылку `tailscale up` печатает в консоль,
подтверждается в браузере на аккаунте пользователя (я не могу пройти этот шаг сам, только человек).
Подробности и следующие шаги (локальный процесс-приёмник задач на ПК, WebSocket для совместной работы,
WebGL-превью в браузере) — в `Docs/Architecture.md`, раздел «Связь VPS ↔ домашний ПК».

## 2026-07-17 — Отказ от входящих соединений к ПК-раннеру: только исходящий опрос

Пробовали входящий SSH (VPS → ПК) как ранний прототип запуска задач для будущей панели. Уткнулись
в реальную проблему: на ПК-раннере постоянно работает AmneziaVPN (нужен пользователю для доступа
к Claude/Anthropic из его региона), и он, похоже, глушит входящие TCP-подключения на Tailscale-интерфейсе
на уровне драйвера — при том что Windows Firewall и служба sshd настроены верно, а сам Tailscale
(другой протокол) пингуется нормально. Пользователь не готов отключать Amnezia даже на секунду теста —
это разорвёт его связь с Claude Code. Решение: не чинить это, а поменять архитектуру — раннер должен
сам инициировать исходящие соединения к VPS (как self-hosted раннер GitHub Actions), а не принимать
входящие. Исходящие уже надёжно работают (иначе не работал бы и сам Claude Code через Amnezia).
Подробности — `Docs/Architecture.md`, раздел «Связь VPS ↔ домашний ПК».
