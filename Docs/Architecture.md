# Архитектура: AI-first Game Dev OS для «GAZON»

Статус: **черновик/roadmap**. Ничего из MCP-серверов и веб-панели ниже ещё не реализовано —
описано, чтобы у обоих разработчиков и у Claude Code была общая карта того, куда мы идём и почему
именно в таком порядке.

## Участники и роли

- Два человека (владелец `gr33njj.dev` + друг), оба используют Claude Code как основного агента.
- Финальный продукт — PC-игра в Steam (не WebGL, не мобильная сборка).
- Веб-панель на `ai.gr33njj.dev` — общий пульт для двоих + агентов: вносить правки, триггерить Unity-билд,
  смотреть/тестировать результат из браузера.

## Почему в таком порядке (фазы)

Полный стек с картинки (Unity MCP + SQLite MCP + Vector DB Docs MCP + Git MCP + веб-панель с билд-фермой)
— это отдельный инфраструктурный проект размером с саму игру. Ставить всё сразу до того, как в Unity
есть хотя бы один рабочий прototip уровня — верный способ утонуть в тулинге и не сделать игру.
Поэтому порядок такой:

### Фаза 0 — Скелет (этот шаг, сделано)
- `Docs/GDD.md` — механики, извлечённые из HTML MVP.
- Структура папок и `.claude/CLAUDE.md`.
- `Database/schema.sql` + `project.sqlite` — сразу в SQLite, без ORM/сервера, просто файл в репозитории.

### Фаза 1 — Unity-проект и порт вертикального среза
- Один из вас создаёт Unity 6 проект (3D URP template) через Unity Hub локально — **этого нельзя сделать
  с текущей машины**, тут нет Unity Editor и GUI. Дальше эта папка `GameProject/` мёржится внутрь
  Unity-проекта (Docs/Design/AI/Database/Tools/.claude — это папки поверх Assets/Packages/ProjectSettings).
- Портируется один цикл: докстанция → полки → выдача → таймер смены. Без QTE/мини-игр/телефона — только
  чтобы убедиться, что цикл ощущается в Unity так же, как в вебе.
- Git-репозиторий на GitHub (см. `.gitignore` — Unity-специфичный).
- Claude Code уже полезен на этом этапе без единого MCP-сервера: обычный доступ к файлам Assets/*.cs
  + Docs/GDD.md как источник истины.

### Фаза 2 — Первые MCP-серверы (по одному, по мере боли)
Строить в порядке, где каждый снимает конкретную повторяющуюся боль, а не потому что «так по схеме»:

1. **Project MCP (SQLite)** — первым, потому что `Database/schema.sql` уже есть и оба разработчика
   уже сейчас руками могут дублировать контент (штрафы/мемы/реплики), если не завести единый источник.
2. **Git MCP** — как только начнутся параллельные ветки между двумя людьми + агентом, чтобы не терять
   изменения друг друга.
3. **Unity MCP (bridge)** — **пересмотрено 2026-07-16**: не пишем свой `HttpListener`-мост с нуля.
   Есть готовый, поддерживаемый, MIT-лицензированный инструмент **MCP for Unity**
   (github.com/CoplayDev/unity-mcp, ex-justinpbarnett/unity-mcp), работает на Unity 2021.3 LTS–6.x,
   ставится через Package Manager → Add from git URL → `https://github.com/CoplayDev/unity-mcp.git?path=/MCPForUnity#main`,
   конфигурируется под Claude Code одной кнопкой (`Window → MCP for Unity → Configure All Detected Clients`).
   Даёт агенту консоль, иерархию сцены, редактирование скриптов, тесты, профайлинг, билды.
   **Важное архитектурное ограничение**: это работает только если Claude Code запущен **на той же
   машине**, где открыт Unity Editor — сессия на VPS (эта) физически не может достучаться до localhost
   на чьём-то домашнем компе. Значит: кто-то из вас ставит Claude Code локально на машину с Unity,
   настраивает MCP for Unity, и именно та (локальная) сессия получает "глаза" на редактор — не эта.
   Раз инструмент уже готов и решает сегодняшнюю боль (ручная пересылка логов консоли туда-сюда),
   имеет смысл ставить его сейчас, а не откладывать до конца Фазы 1.
4. **Docs MCP (semantic search)** — когда `Docs/` и `Design/` разрастутся настолько, что grep перестаёт
   справляться (обычно это происходит не раньше, чем в Design/ появится десяток файлов).

### Фаза 3 — Веб-панель `ai.gr33njj.dev`

Решено 2026-07-16, конкретный порядок:
1. **Выделенная машина-раннер** — компьютер, на котором постоянно включён Unity Editor с открытым
   проектом (нужен для MCP for Unity, см. `AI/MCP.md` — это расширение редактора, не headless-режим).
2. **GitHub Issues/Projects** — общий трекер задач для двоих, бесплатно и без своей инфраструктуры,
   пока нет самой панели.
3. **Сама панель** — тонкий фронтенд поверх раннера: задача из панели → GitHub issue/ветка →
   раннер (локальный Claude Code + MCP for Unity) забирает и выполняет → пушит результат →
   панель показывает статус. Строить только после того, как пункты 1–2 реально используются
   и понятно, чего именно панели не хватает поверх голого GitHub.

Это отдельный сервис, не часть Unity-проекта.

#### Связь VPS ↔ домашний ПК (решено 2026-07-16, частично сделано)

Комп-раннер сидит дома за NAT без белого IP — VPS не может достучаться к нему напрямую, а пробрасывать
порт на роутере (port forwarding) сознательно не делаем: это дыра в домашнюю сеть и ломается при смене
IP у провайдера. Вместо этого — **Tailscale**: приватная mesh-сеть поверх WireGuard, оба узла инициируют
исходящее соединение наружу (это всегда проходит через NAT без настройки роутера), Tailscale их сводит.
После этого VPS обращается к домашнему ПК по его Tailscale-адресу так, будто они в одной локальной сети.

- **Статус: сделано 2026-07-17.** Оба узла в одном тейлнете (аккаунт jj.gr33nrec@):
  - VPS: `gr33njj-vps`, `100.87.134.21` (Linux; поставлен через `curl -fsSL https://tailscale.com/install.sh | sh` + `tailscale up`)
  - ПК-раннер: `mddt-k508-1`, `100.110.129.50` (Windows)
  С VPS сейчас можно достучаться до ПК напрямую по `100.110.129.50` (и наоборот) — проверено
  `tailscale ping mddt-k508-1`, есть даже прямое P2P-соединение в обход relay-сервера DERP, не только
  через него (быстрее и надёжнее). Если прямое соединение отвалится (ПК за жёстким NAT/файрволом),
  Tailscale сам переключится на DERP-relay — тоже работает, просто чуть медленнее.
- **Важная поправка 2026-07-17, входящие соединения к ПК отменяются.** Пробовали проверить входящий SSH
  от VPS к ПК (ранний прототип будущего запуска задач) — и уткнулись в реальную стену: на ПК-раннере
  постоянно включён **AmneziaVPN** (полнотуннельный VPN, нужен пользователю, чтобы вообще достучаться
  до Claude/Anthropic из его региона). Он, судя по всему, перехватывает/роняет входящие TCP-соединения
  на Tailscale-интерфейсе на уровне драйвера — при этом `tailscale ping` работает (это отдельный протокол
  самого Tailscale, не обычный TCP), а SSH-хендшейк по 22 порту не проходит, хотя Windows Firewall и
  сервис sshd настроены правильно (проверено: правило разрешает Private-профиль, интерфейс Tailscale
  классифицирован как Private). Отключать Amnezia ради теста пользователь не готов — потеряется связь
  с Claude Code. Чинить это не стали (не стоит риска и времени ради прототипа).
- **Вывод: раннер должен САМ стучаться наружу к VPS, а не наоборот.** Тот же паттерн, что у self-hosted
  раннеров GitHub Actions — исходящие соединения от ПК уже надёжно работают (через тот же Amnezia+Tailscale,
  раз Claude Code вообще подключается), поэтому вся будущая связь панель↔раннер должна строиться как:
  небольшой процесс на ПК периодически сам опрашивает VPS (или держит исходящий WebSocket) на предмет
  новых задач → выполняет через локальный Claude Code CLI (`claude -p "..."`) → сам отправляет результат
  обратно на VPS. Ничего не должно пытаться достучаться ДО ПК первым. Этот процесс ещё не написан —
  часть будущей работы над самой панелью (пункт 3 выше).

#### Совместная работа вдвоём в реальном времени

Обычная задача для веб-приложения, не специфика этого проекта — WebSocket или Server-Sent Events,
которые пушат статус задачи/лог сборки во все открытые вкладки браузера. У сайта `gr33njj.dev` уже есть
Next.js + pm2 (см. память `gr33njj-dev-deploy`) — тот же стек подходит и для панели, ничего нового
осваивать не нужно. Не требует раннера/Tailscale — это чисто фронтенд-бэкенд часть на самой VPS.

#### Превью билда прямо в браузере

Unity умеет собирать **WebGL-версию отдельно от Steam-сборки** только ради предпросмотра изменений —
такой билд открывается прямо в `<canvas>` без установки, как игры на itch.io. Практическая схема:
раннер после обработки задачи собирает WebGL-билд (нужен модуль **WebGL Build Support**, ставится
отдельно через Unity Hub), результат заливается на VPS (через тот же Tailscale-канал), панель отдаёт
его как статическую страницу. Ограничения: сборка занимает минуты, не секунды (не «живой» хот-релоад);
WebGL — не побайтово то же самое, что билд для Steam (нет файловой системы, отличается производительность),
для масштаба этой игры это не проблема. Релизная сборка для Steam остаётся отдельным Windows/Mac/Linux
билдом, собираемым по явному запросу, а не при каждой задаче.

**Статус 2026-07-17, сделано:**
- GitHub Actions self-hosted runner зарегистрирован на ПК-раннере (`mddt-k508-1-runner`), запускается
  вручную через `.\run.cmd` в `C:\actions-runner` (постоянная служба Windows не завелась через
  `RunnerService.exe install` — вылезла ошибка «нельзя стартовать службу из командной строки»,
  разбираться с `installutil`/ServiceController не стали, пока работает через открытое окно консоли —
  этого достаточно для проверки схемы, автозапуск можно донастроить отдельно позже).
- `Assets/Editor/BuildScript.cs` — сборка WebGL и через меню (`Gazon → Собрать WebGL-превью`), и через
  `-executeMethod Gazon.EditorTools.BuildScript.BuildWebGLCI` для CI (отдельный метод — возвращает код
  выхода процессу, чтобы GitHub Actions понимал успех/провал).
- `.github/workflows/webgl-build.yml` — ручной запуск (`workflow_dispatch`), собирает WebGL на раннере,
  результат кладёт как GitHub Actions artifact.
- **Доставка на VPS — не через SSH/scp, а через HTTPS.** Пробовали PC → VPS по SSH (порт 22) — тот же
  AmneziaVPN на раннере рвёт исходящий SSH (мгновенный `Permission denied` на Tailscale-адрес, зависание
  на публичном IP), при этом HTTPS через тот же VPN работает нормально (иначе не работал бы сам Claude
  Code). Решение: на VPS поднят маленький Node-сервис `/opt/gazon-upload` (systemd `gazon-upload.service`,
  слушает `127.0.0.1:8899`, только пользователь `gazon-deploy`, без root) — принимает zip по `PUT` с
  bearer-токеном (GitHub secret `GAZON_UPLOAD_TOKEN`), распаковывает в `/var/www/gazon-preview`. nginx
  проксирует `https://ai.gr33njj.dev/upload`. Workflow после сборки: `Compress-Archive` → `curl.exe -X PUT`.
  Попутные грабли: `unzip` возвращает код `1` (не 0) при предупреждениях о бэкслешах в путях из
  Windows-архивов — это «успех с предупреждением», не провал; `execFileSync` кидает на любой ненулевой
  код, пришлось переключиться на `spawnSync` с ручной проверкой (`status >= 2` — настоящая ошибка).
  Также Unity по умолчанию сжимает WebGL-билд в Brotli (`.wasm.br`/`.data.br`/`.js.br`) — без
  `Content-Encoding: br` в nginx браузер не распакует файлы и игра зависнет на progress bar.
- **Живой пример**: https://ai.gr33njj.dev — первая рабочая страница с играбельным WebGL-превью,
  залитая полностью автоматически из workflow (не руками).
- **2026-07-17, первая успешная сборка end-to-end** (run [29585024189](https://github.com/gr33njj/gazon/actions/runs/29585024189),
  56 сек, артефакт `webgl-preview` ~4.9 МБ). По пути нашли и починили реальный и не самый очевидный баг:
  PowerShell-вызов `& "Unity.exe" ...` возвращал управление через ~2 сек после старта (когда закрывался
  поток STDOUT/STDERR), хотя Unity ещё активно работал через дочерние процессы (`bee_backend.exe`,
  `netcorerun.exe` — компиляция скриптов/сборка WebGL). Раннер считал шаг завершённым и убивал всё
  дерево процессов Unity как «осиротевшее» — отсюда падения на разных стадиях в разных попытках, без
  единой понятной причины. Нашли через `_diag/Worker_*.log` раннера (`ProcessInvokerWrapper` явно
  логирует "STDOUT/STDERR stream read finished" и последующее убийство дерева процессов). Исправлено
  через `Start-Process -Wait -PassThru` — ждёт реального завершения по process handle, а не по потоку.
  Заодно по дороге чинили: `pwsh` не установлен на раннере (нужен обычный `powershell`), Unity Personal
  license конфликтовал с одновременно открытым интерактивным редактором, и WebGL-модуль в Unity 6.5
  называется «Web Build Support», а не «WebGL Build Support».

Оба пункта выше подтверждены первым успешным прогоном — модуль стоит, сцена в Build Settings есть.

- Авторизация панели — минимум базовая (это приватный инструмент для двоих, не публичный сервис).
- Самая рискованная по времени часть всей схемы — не сама панель (обычный веб-стек), а надёжность
  связки раннер-процесс + Tailscale + Unity batch/WebGL-сборка. Не начинать до того, как раннер и
  GitHub Issues (пункты 1–2 выше) реально в ходу.

## Ограничения текущей машины

Эта VPS (`gr33njj.dev`, Linux) — веб-сервер для сайта-визитки, тут нет Unity Editor и не будет
(headless Unity build на Linux технически возможен, но лицензирование и GPU-рендеринг для превью
на этой машине не рассматриваются, пока это не станет отдельным осознанным решением с ценой ресурсов).
Практический вывод: Unity-разработка идёт на ваших локальных машинах, эта VPS в лучшем случае
может участвовать как хост для веб-панели (фаза 3) и для лёгких MCP-серверов (Project/Docs/Git),
которые не требуют самого Unity Editor.
